# 需求冻结: 仓库激进瘦身与架构收敛

- 日期: 2026-04-04
- 任务入口: 用户要求基于当前最新仓库，理解项目架构后进行激进精简，删除过期、无用、重复或可再生成的文档、资源、脚本，并保持高内聚、低耦合

## 目标

在不破坏当前 `v2.3.56` 架构闭环、安装/验证主流程、发布入口和测试基线的前提下，对仓库执行一次高强度瘦身，优先消除以下问题：

- 历史治理文档、计划文档、状态文档无限堆积
- 已退役镜像或兼容面继续占据主仓体量
- 可再生成的分发/输出资产与源码长期并存
- 明显重复的脚本入口和缺少消费者的遗留脚本
- 目录职责边界模糊，source of truth 与 mirror/generated surface 混杂

## 交付物

- 一份基于真实引用关系和职责边界的精简实施结果
- 删除后的目录/文件收敛结果
- 必要的说明文档更新，反映新的 source of truth、保留面与可再生成面
- 通过定向验证的可提交变更

## 删除准入标准

只有满足以下至少一类条件的内容才允许删除：

1. 明确为生成物，且可由现有源码、脚本或发布流程稳定重建
2. 明确为历史阶段性文档，且不再作为当前 operator / contributor / runtime / release 的权威入口
3. 明确为重复镜像或重复资源，且当前主线已经收敛到更小的权威面
4. 明确无消费者引用，且不属于发布、安装、验证、回放或法律披露必须保留面
5. 明确与当前架构闭环冲突，属于已退役实现或仅为旧拓扑保留的叙事负担

## 必须保留的边界

- 当前主线安装、检查、卸载与 CLI 主入口
- `packages/**` 的 package-owned 语义核心
- 当前 release / runtime / adapter / contract 所需的权威配置与测试
- 法务与第三方披露必须保留面
- 当前仍被测试、构建、安装、发布流程消费的目录

## 验收标准

- 删除后的仓库仍能通过关键定向测试与完整性检查
- README / CONTRIBUTING / 架构文档中不再将已删除内容描述为当前权威入口
- 精简结果能清楚解释“为什么删、为什么保留、谁是新的权威面”
- 工作树整洁，无临时分析产物残留

## 非目标

- 不在本轮内重写整套发布系统
- 不无证据地删除当前测试仍在消费的文件
- 不为了体量而牺牲安装、验证、发布、法务披露的完整性
